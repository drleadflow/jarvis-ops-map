What Is This

What Is Jarvis Command Center

The complete Jarvis product: a Next.js 49-page operations dashboard (forked from Blade Command Center) connected to an always-on AI agent runtime. Agents execute tasks autonomously with heartbeats, budgets, and approval gates. Not a chatbot — a foreman with workers, memory, and verification.

Infrastructure

Three Railway Services

The system runs on 3 Railway services:

  • command-center (Next.js) — 49-page dashboard at command-center-production-34b8.up.railway.app. Handles UI, auth, API proxying. Built with nixpacks.
  • jarvis-runtime (Node.js supervisor) — Always-on agent runtime at jarvis-runtime-production.up.railway.app. Runs the fleet manager, heartbeat scheduler, budget enforcer, approval gate. Serves Paperclip-compatible API.
  • jarvis-bot (Telegram) — Telegram bot interface at jarvis-bot-production-fbba.up.railway.app. Webhook mode with grammy.

7 Layers

7-Layer Architecture

  • Layer 7: Self-Learning — Pattern extraction, memory, improvement loop. The system learns from outcomes and stores corrections, patterns, and reflections.
  • Layer 6: Verification — Tests, lint, screenshots, schema checks. Echo gate provides blocking quality control before any merge or promotion.
  • Layer 5: Policy — Risk lanes (Safe / Ask-Once / Human-Only), budget tracking, acquisition protocol, rollback classification.
  • Layer 4: State Engine — Supabase: projects, tasks, checkpoints, artifacts, approvals, runs, memory, budgets.
  • Layer 3: Workers — Builder, Researcher, Tester, Deployer, Debugger. Each inherits from BaseWorker with tool access and heartbeat.
  • Layer 2: Supervisor — Alpha brain: decompose task graphs, delegate to fleet, verify results, report back. Foreman constraints and auto-pause.
  • Layer 1: Communication — Telegram bot, Claude Code CLI, Dashboard UI, Email (IMAP + nodemailer).

Request Flow

Request Flow

How a task moves through the system from input to completion:

  • 1. User sends task (Telegram / Dashboard / Claude Code)
  • 2. Supervisor decomposes into task graph with dependencies
  • 3. Policy engine checks risk lane + budget availability
  • 4. Fleet manager assigns to available agent based on role match
  • 5. Agent executes with tool access, heartbeating every 30s
  • 6. Verification pipeline validates result (Echo cheap + heavy gates)
  • 7. Memory engine extracts patterns from outcome
  • 8. Report back to user via original channel

State Machine

Task Lifecycle

Every task follows this state machine:

  • proposed → accepted → in_progress → verification → done → archived
  • in_progress → blocked → (awaiting_human | awaiting_dependency) → in_progress
  • accepted → rejected (bounced back with reason)

Max task duration: 30 min per agent. Stale threshold: untouched 2 cycles. Blocked escalation: blocked >1 cycle triggers Head of Operations.

NATO Fleet

NATO Manager Fleet

Five persistent managers using NATO phonetic alphabet. Each manager spawns numbered sub-agents and uses git worktree isolation:

  • Alpha — Deploy & Infrastructure Manager — Docker, CI/CD, Railway, env validation, healthchecks. Branch: alpha/{task-slug}
  • Bravo — Frontend & Dashboard Manager — Dashboard components, UI testing, frontend coverage. Branch: bravo/{task-slug}
  • Charlie — E2E Test Manager — End-to-end test infrastructure, critical flow tests. Branch: charlie/{task-slug}
  • Delta — Integration Test Manager — Cross-package integration, data flow validation. Branch: delta/{task-slug}
  • Echo — Verification & Quality Gate — Final verification sweeps, quality enforcement. BLOCKING gate — FAIL = no merge. Reads across all worktrees but does NOT write.

Workers

Worker Agents

Six specialized workers that execute tasks assigned by managers:

  • Morning Briefing — Daily briefing agent (6AM cron). Pulls Meta Ads, leads, content, tasks. Delivers summary to Telegram.
  • Ad Monitor — Ad & competitor monitor (every 4 hours). Tracks changes in competitor ads and landing pages.
  • Content Pipeline — Content pipeline agent (9AM weekdays). Reviews calendar, drafts content for gaps.
  • Jarvis — Core task decomposition agent. Routes and delegates work to appropriate workers.
  • Builder — Code generation worker. Writes code, creates files, implements features.
  • Researcher — Research worker. Web search, docs, codebase analysis. Read-only.

Lifecycle

Agent Lifecycle

Every agent follows this lifecycle with automatic state management:

  • idle → running (via wake command or schedule trigger)
  • running → heartbeat every 30 seconds while active
  • running → idle (auto-pause after 3 consecutive idle heartbeats)
  • idle → running (wake again via API or schedule)

Budget enforced per agent with daily caps. Approval required for Lane 2/3 actions. Fleet manager tracks all agent states in Supabase agents table.

Budget

Budget Enforcement

BudgetEnforcer tracks token usage per agent with configurable daily caps. When an agent exceeds its budget:

  • Agent is auto-paused
  • Alert fires (Medium severity)
  • Head of Operations is notified
  • Budget resets at midnight UTC
▼
Approval

Approval Gates

Lane 2/3 actions trigger an approval flow:

  • Task pauses and enters awaiting_human state
  • Telegram inline keyboard sent to admin (Approve / Reject / Escalate)
  • 30-minute timeout — task marked awaiting_human, system moves to next task
  • Max 2 interrupts per task stage; remaining questions batched
▼
Rollback

Rollback Classification

Every mutating action is classified for rollback capability:

  • Fully reversible → Manager approval sufficient
  • Conditionally reversible → Head of Operations approval required
  • Irreversible → Human approval + Tier 2 security review required

Alerts

Alert Conditions (8 Rules)

  • Echo override spike (High) — 3+ overrides on same check in 7 days
  • Rollback failure (Critical) — Any rollback that doesn't fully succeed
  • Blocked-human pileup (High) — 5+ tasks awaiting_human simultaneously
  • Acquisition cost explosion (Medium) — Single acquisition exceeds 2x budget ceiling
  • Stale queue (Medium) — 10+ tasks untouched for 2+ cycles
  • Low-confidence action (High) — Auto-promote attempt with confidence < Medium
  • Budget breach (Medium) — Any agent exceeds token budget
  • Merge conflict storm (Medium) — 3+ conflicts in single merge cycle
